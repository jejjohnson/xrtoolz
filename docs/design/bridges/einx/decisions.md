# xrtoolz.einx — Decisions

> **Implementation note (v0.2.0).** The bridge is implemented under
> `src/xrtoolz/einx/`. One decision changed during implementation: **D9
> is reversed** — einx is now a **core runtime dependency**, not an
> optional `[einx]` extra (see D9 below). This unblocks using einx in
> core kernels across the package. `import xrtoolz` still does not pull
> einx eagerly; the bridge imports einx lazily inside its Layer-0
> functions, so the cost is paid only when the bridge is used. Open
> question O5 is resolved: the documented alias is `xnx`.

## Resolved

| # | Question                                              | Resolution                                                                                                   |
| - | ----------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| 1 | Module name                                           | `xrtoolz.einx`. Matches the upstream library name; readers immediately know what the bridge wraps.            |
| 2 | Carrier                                               | `DataArray`-first; `Dataset` via explicit `pack_dataset` / `unpack_dataset`.                                  |
| 3 | Pattern syntax                                        | Exactly einx's. No vendored grammar; we only validate dim names.                                              |
| 4 | Backend                                               | Whatever the input DataArray wraps (numpy / dask / jax / torch).                                              |
| 5 | Coord forwarding                                      | Surviving dims keep the first input's coord; reduced dims drop coords; new dims need `coords=` override.      |
| 6 | Coord-mismatch policy                                 | Default raise; opt-in `align=True` for xarray-style realignment.                                              |
| 7 | Operator wrappers                                     | One per Layer-0 function, generated by hand for `__init__` clarity (no metaclass magic).                      |
| 8 | Signature inference                                   | Bridge parses the pattern itself for dim-name extraction; defers einx semantics to runtime.                   |
| 9 | Install                                               | `pip install xrtoolz[einx]` adds einx to deps. Lazy import.                                                   |

## Key design choices

### D1 — Pattern axes refer to dim names, not positions

einx itself is position-based by default — its pattern's axis tokens
match the input array's axes by order. The bridge inverts this: the
tokens are *dim names*, and we transpose to match einx's positional
expectation before dispatch.

The cost is one extra `.transpose(...)` per input (cheap, since
xarray's transpose is a view). The benefit is that patterns become
order-independent — you can write `"time lat lon"` regardless of how
the upstream pipeline ordered the dims, and a refactor that touches
dim order never breaks an einx call.

### D2 — Bridge does not invent dim sizes

If a pattern axis has no source for its size (not on any input, not
in `shape_kwargs`, not in `coords`), we raise. This is the same
policy as einx itself for axis-binding; we propagate it cleanly
through the labeled layer so error messages reference dim names
rather than positional axis indices.

### D3 — Coords on dropped dims disappear silently

Reductions drop the dim, including its coord. No warning. xarray
does the same (`da.sum(dim="time")` doesn't warn about the lost
`time` coord), and warning here would be noise for every
reduction-shaped op.

### D4 — Coords on new dims are caller-supplied

When a pattern creates a new dim — `lat -> month lat`, `month=12`,
say — the output has no coord for `month`. The caller must pass
`coords={"month": ...}` if they want one. The alternative (autogen
range index 0..N-1) would silently produce wrong coord semantics in
half the cases.

### D5 — `align=` is opt-in, not default

xarray's default is to realign by coord; numpy and einx's default
is to combine by position. The bridge follows numpy/einx, because
realignment changes the *shape* of the output in a way that's hard
to predict from a pattern. Users who want xarray semantics flip
`align=True` per call; users who treat DataArrays as labeled tensors
get strict failure on misalignment.

### D6 — No Operator that wraps `pack_dataset`

`pack_dataset` and `unpack_dataset` are utility xarray functions, not
math operations. Wrapping them as `Operator`s would invite users to
sprinkle them into pipelines instead of designing the pipeline around
a single packing point. They stay as plain functions.

### D7 — `matmul` / `outer` / `batch_matmul` exist as convenience

These could be one-line einsums. Carrying named convenience functions
costs almost nothing (~30 lines) and pays off in notebook readability.
They're explicitly documented as *thin wrappers over `einsum`*, so
users never feel they have to learn two surfaces.

### D8 — No "ambiguous-axis" detection

If a pattern reuses the same axis name across multiple inputs (the
contraction case in `einsum`), that's fine and follows einx. If a
single input has duplicate dim names (which xarray itself forbids),
xarray raises before we ever see the array.

### D9 — `einx` is a core dependency *(reversed v0.2.0)*

**Original decision:** keep einx behind an optional `[einx]` extra so a
plain `pip install xrtoolz` doesn't pull it.

**Reversed:** einx is now a **core runtime dependency**. The motivation
is the repo-wide adoption of einx in core kernels (metrics, transforms,
interpolate, …) — those modules cannot depend on an optional extra. einx
pulls only numpy by default, so the install cost is near-zero, which is
what made the original extra "near-zero cost" anyway. `import xrtoolz`
stays light because the `xrtoolz.einx` submodule is not imported at
package init and imports einx lazily inside its Layer-0 functions.

## Open

| # | Question                                                                                                       | Status                                                                                                                          |
| - | -------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| O1 | Should `Einsum` accept Dataset inputs via implicit `pack_dataset`?                                              | Lean **no** — explicit pack/unpack keeps the contract clean. Revisit once usage data is in.                                     |
| O2 | Should `align=True` use xarray's inner join, outer join, or a `join=` kwarg?                                    | Default to xarray's `xr.align(..., join='exact')` semantics; expose `join=` if usage demands.                                   |
| O3 | Do we need a `vmap`-style operator that re-runs an inner op per slice along a named dim?                        | Probably belongs in `xrtoolz.combinators` (companion to `ApplyToEach`), not `xrtoolz.einx`.                                     |
| O4 | Should string ops (`op="median"`) dispatch through `da.reduce` for dask-compat, or through einx's own reducers? | Prefer einx for consistency; need to verify dask path doesn't lose laziness.                                                    |
| O5 | Naming: should the alias be `xnx` or `xex` or `xeinx`?                                                          | Pick one and document it. `xnx` reads cleanest; finalise on first PR.                                                            |

## Rejected

| # | Option                                                                              | Why rejected                                                                                                                     |
| - | ----------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| R1 | Custom DSL that wraps einx's syntax with xarray-isms                                  | Two surfaces to learn instead of one. einx is already excellent.                                                                  |
| R2 | Auto-broadcast (xarray-style) by default                                              | Hides shape errors. xarray users who want this can call `.broadcast_like(...)` first; everyone else gets strict shapes.            |
| R3 | Treating `einx` as part of `xrtoolz.transforms`                                       | einx is a *backend dependency*, not a transform family. Same logic that put `inference` next to `geo` rather than inside it.       |
| R4 | Auto-generated `Coord` for new dims (e.g. `np.arange(N)`)                              | Silently wrong half the time. Forcing explicit `coords=` makes the contract obvious.                                              |
